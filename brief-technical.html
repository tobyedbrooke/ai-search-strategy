Technical Team Brief

Make Trade Republic readable by AI

Three technical changes that determine whether AI systems can read and cite our content. All are short-term. None require new content — only how we serve what already exists.

Why this matters. AI systems like ChatGPT, Perplexity and Google AI Overviews fetch pages the same way a simple HTTP client does — they do not execute JavaScript. If a page's content is injected by JS after load, the AI sees an empty shell. Trade Republic's Help Centre is currently in this state. Additionally, without structured data markup (schema), AI has to guess what TR is, what products cost, and which text answers which question. These three changes fix both problems.
1
Audit CSR vs SSR across Help Centre and main site Short term

Before migrating anything, confirm which pages are actually client-side rendered (invisible to AI) and which are server-side rendered (readable). This is a 30-minute audit.

How to check:

  • 1.In Chrome, go to support.traderepublic.com and any Help Centre article. Press ⌘+U (View Source). If the body contains your article text, it's SSR. If you see mostly empty <div> tags and script imports, it's CSR.
  • 2.Confirm with curl — this is exactly what AI crawlers see:
curl -s https://support.traderepublic.com/en-de/articles/your-article-slug | grep -c "Trade Republic"

If the count is 0 or very low, AI crawlers see nothing. If it returns meaningful paragraph text, it's crawlable.

Also check the main site — traderepublic.com/en-de product pages (Interest Account, ETF Savings Plan). The homepage stats and feature descriptions may be JS-rendered even if they appear visually. View Source and check.

Output: A list of pages confirmed as CSR (need migration) vs SSR (already crawlable). This scopes the migration work precisely.

2
Migrate Help Centre from CSR to SSR Short term

If the audit confirms CSR, migrate Help Centre to server-side or statically rendered HTML. The goal: the full article body must be present in the HTTP response before any JavaScript executes.

Options (in order of preference):

  • A.Static Site Generation (SSG) — pre-build all Help Centre articles as static HTML at deploy time. Best for content that doesn't change per-user. Zero runtime cost. Recommended if the HC content is served from a CMS or structured source.
  • B.Server-Side Rendering (SSR) — render the page on the server for each request. Necessary if articles are personalised or dynamically filtered. Slightly more infrastructure overhead.
  • C.Targeted static injection — if a full migration is too complex short-term, at minimum inject the article title and first 3 paragraphs as static HTML in the <body>, before the JS bundle loads. Imperfect but gets critical content into the crawl.

How to verify it worked: Re-run the curl command from Task 1 after the change. Article body text should now appear in the raw response. Also run the URL through Google's Rich Results Test — if it can parse the page content, AI crawlers can too.

3
Add FAQPage schema to the top 20 Help Centre articles Short term

FAQPage JSON-LD tells AI systems exactly which text answers which question. It feeds directly into Google AI Overviews and is used by Perplexity to extract Q&A pairs. Currently there is zero schema markup anywhere on traderepublic.com.

Implementation: Add a <script type="application/ld+json"> block to the <head> of each Help Centre article. Structure:

{ "@context": "https://schema.org", "@type": "FAQPage", "mainEntity": [{ "@type": "Question", "name": "Is Trade Republic safe?", "acceptedAnswer": { "@type": "Answer", "text": "Trade Republic Bank GmbH is regulated by BaFin and the Deutsche Bundesbank, holds an ECB banking licence, and provides €100,000 deposit protection under the German deposit guarantee scheme. Customer securities are held in custody at Clearstream." } }] }

Prioritise articles that answer: "Is TR safe?", "What are TR's fees?", "How does TR make money?", "What is TR's interest rate?", "How do I contact TR support?", "How do I close my account?" — these are the highest-frequency AI queries about TR.

Key rule: The answer text in schema must match visible page content exactly. Google and AI systems cross-reference the two.

4
Add Organisation and Product schema to homepage and product pages Medium term

Organization schema on the homepage establishes TR's identity — name, regulation, founding date, markets, licence — as a machine-readable fact. Product schema on the three core product pages gives AI exact current rates and fees to quote directly.

Homepage — Organization schema:

{ "@context": "https://schema.org", "@type": "FinancialService", "name": "Trade Republic", "legalName": "Trade Republic Bank GmbH", "url": "https://www.traderepublic.com", "foundingDate": "2015", "numberOfEmployees": { "@type": "QuantitativeValue", "value": 1200 }, "areaServed": ["DE","AT","FR","ES","IT","NL","BE","PT","FI","..."], "regulatoryBody": { "@type": "Organization", "name": "BaFin" } }

Product pages to add Product schema:

  • →ETF Savings Plan — fee: €0, minimum: €1/month
  • →Interest Account — current rate: 2.00% p.a. (update this whenever rate changes)
  • →Stock/ETF trading — fee: €1 per trade

Important: Product schema rates must be kept current. A stale rate in schema is worse than no schema — AI will cite the wrong number with confidence.

Expected outcome After all four tasks: Help Centre articles appear in Google AI Overviews for TR-specific questions. Perplexity begins citing TR's own pages rather than third-party review sites for factual queries. Organisation and Product schema means AI quotes current TR rates accurately rather than inferring from stale editorial content.
